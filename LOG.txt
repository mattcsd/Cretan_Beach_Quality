        QA:
        1. Fixed height in CurrentWeatherView

    You said you don’t know how to fix it.
    Solution: Remove the fixed height constraint and let Auto Layout calculate the height based on the content. The view already has top and bottom constraints from the labels. The timeLabel is pinned to the bottom, and weatherIcon is pinned near the top. Auto Layout can then determine the natural height. You would also need to ensure the container view doesn't have a fixed height; instead, let it be determined by its subviews.

    2. Priority 999 – what is it?

    Layout priority ranges from 1 to 1000. 1000 is required. 
    999 means “almost required – satisfy if possible, but if 
    it conflicts with a required constraint, break this one”. 
    It’s used to allow the hourly forecast view to determine its 
    own height while still having a bottom anchor to the container. 
    If the content height would make the cell too big, the bottom 
    constraint can be ignored. A more readable alternative 
    is .defaultHigh (which is 750) or .required (1000). 999 is a 
    bit arbitrary.

    3. Why reload the whole table in onWeatherLoaded?

    Because the number of forecast days might change 
    (though usually it’s fixed) and all cells depend on weather data. 
    you could reload just the section, but reloading the whole table 
    is simpler and fine for a small table.

    4. viewDidLayoutSubviews for the header

    When you set tableView.tableHeaderView, the table does not 
    automatically adjust the header’s height if the content inside 
    changes (e.g., due to dynamic font sizes or rotation). 
    You have to manually calculate the fitting size and assign 
    the new height. That’s what the code does.

    5. toggleExpanded calling onWeatherLoaded

    As explained, it forces the table to reload so the cell 
    can show/hide the hourly view. Without it, the UI would 
    not update until another event triggered a reload.

    6. Hard‑coded geonamesUsername

    For a test app it’s fine. In production, you would store 
    API keys in a configuration file that is not committed to 
    version control (e.g., .xcconfig, Info.plist, or environment 
        variables). You could also use a secure service like AWS 
    Secrets Manager. But never hard‑code secrets.

    7. Returning item from handleBeachSelection is redundant

    You are correct that the item does not contain the coordinates. 
    But the caller (didSelectRowAt) already has the item from 
    viewModel.beach(at: indexPath.row) before calling 
    handleBeachSelection. So you are passing back the same item that 
    the caller already has. That’s the redundancy. You could 
    change handleBeachSelection to return only the coordinates 
    (or throw), and let the caller use its own item. That reduces 
    unnecessary data passing.

    8. Difference between async function and returning a Task

    async function: marks a function that can suspend. When you call it, 
    you must use await. The function runs on the same thread and resumes 
    on the same thread after suspension. It does not create a new task.
    Task { }: creates a new top‑level task that runs concurrently. 
    You use it to call async code from a non‑async context (e.g., from 
    viewDidLoad). It returns a Task object that you can cancel or await.
    In ViewModel.loadBeaches(), you put Task { } inside the method. 
    That means the method returns immediately, and the network work 
    happens in the background. The caller cannot await its completion. 
    That’s why you still use callbacks (onLoadingChanged, onDataUpdated) 
    to notify the ViewController. If you made loadBeaches itself async, 
    then the ViewController would have to call await viewModel.loadBeaches() 
    inside a Task, which is also fine – but then you would not need the 
    callbacks for loading state because you could update UI directly 
    after the await. Your current design mixes both patterns, which is 
    a bit confusing.

    9. Middleman in navigation

    Your supervisors said that creating the DetailViewModel inside the 
    ViewController is an extra middleman. The alternative is to have 
    DetailViewController create its own ViewModel from raw data (beach 
    item, latitude, longitude). Then the ViewController only passes 
    raw data and doesn't know about DetailViewModel. This reduces 
    coupling and makes the ViewController simpler.

    10. DispatchQueue.main.async vs await

    DispatchQueue.main.async schedules a block of code to run later on 
    the main thread. You have to manually ensure you’re on the main 
    thread for UI updates.await inside an async function 
    tells the system to suspend the function while waiting for an 
    asynchronous operation. When the operation completes, the function 
    resumes on the same thread (typically the main thread if that’s 
    where it was called). No manual dispatch needed, and the code 
    looks linear.
